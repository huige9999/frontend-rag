# 37. 提示词工程 - 提示词评估

## 课程概述

本课时介绍了 Promptfoo 这一专业的提示词评估工具，帮助开发者系统地测试、比较和优化提示词的质量。

## 核心工具：Promptfoo

### 简介
Promptfoo 是一个专为提示词工程而设计的开源测试与评估工具，官方网站：https://www.promptfoo.dev/

### 核心功能
1. **多提示词对比**：比较多个提示词在相同输入下的输出质量
2. **多模型对比**：比较 GPT-4、Claude、Gemini、LLaMA 等模型对同一提示词的输出差异
3. **批量输入测试**：用多个输入样例评估提示词的稳定性
4. **手动或自动评分**：支持人工打分、关键词匹配、甚至调用 LLM 对输出进行自评
5. **报告输出与分析**：自动生成 Markdown、HTML、JSON 格式的测试报告
6. **命令行 & Web UI**：提供 CLI 工具，也支持运行 `promptfoo web` 启动可视化评测界面
7. **CI 集成**：可集成至 GitHub Actions、CI/CD 流程，做提示词的"自动回归测试"

## 快速开始

## 配置文件详解

### 基本配置结构
```yaml
description: "My eval"  # 评估描述

prompts:               # 提示词模板
  - "提示词1"
  - "提示词2"

providers:             # 模型提供者
  - "openai:gpt-4o-mini"
  - "openai:gpt-4o"

tests:                 # 测试输入
  - vars:
      topic: "测试主题"

eval:                  # 评估方式
  automatic:
    - type: keyword-match
```

### 配置示例一：基础模板
```yaml
description: "My eval"

prompts:
  - "Write a tweet about {{topic}}"
  - "Write a concise, funny tweet about {{topic}}"

providers:
  - "openai:gpt-4o-mini"
  - "openai:gpt-4o"

tests:
  - vars:
      topic: bananas

  - vars:
      topic: avocado toast
    assert:
      - type: icontains
        value: avocado

      - type: javascript
        value: 1 / (output.length + 1)

  - vars:
      topic: new york city
    assert:
      - type: llm-rubric
        value: ensure that the output is funny
```

### 配置示例二：中文总结任务
```yaml
prompts:
  - id: basic-summary
    label: "基础总结"
    raw: "请用一句话总结下面这段话：{{input}}"

  - id: formal-summary
    label: "正式语气总结"
    raw: "请用正式、简洁的语气总结以下内容：{{input}}"

providers:
  - id: ollama:llama3
  - id: ollama:gemma3

tests:
  - vars:
      input: "小明今天早上迟到了，因为他错过了地铁。"
  - vars:
      input: "天气预报说今天有雨，所以我们取消了郊游计划。"
  - vars:
      input: "这家咖啡馆的氛围很好，适合工作和学习。"

eval:
  automatic:
    - type: keyword-match
      criteria:
        - "总结"
        - "简洁"
```

### 配置示例三：招聘广告生成
```yaml
prompts:
  - id: good-job-title
    label: "好提示词：具体要求 + 输出格式"
    raw: |
      请根据以下岗位信息，撰写一句吸引人的招聘广告标题：
      - 以年轻人喜欢的口吻
      - 控制在15个字以内
      - 用一句话概括岗位亮点
      岗位信息：
      {{input}}

  - id: bad-job-title
    label: "坏提示词：无要求，模糊指令"
    raw: |
      写一句话。

providers:
  - id: ollama:llama3

tests:
  - vars:
      input: "我们是一家AI初创公司，正在招聘前端开发，技术栈为Vue和TypeScript，提供远程办公和股票期权。"
  - vars:
      input: "本公司招聘客服专员，工作环境轻松，带薪培训，有下午茶。"
  - vars:
      input: "招聘Python数据工程师，参与大模型推理平台建设，年薪40万起。"

eval:
  automatic:
    - type: llm-rubric
      rubric: |
        请从以下维度为这个输出打分（1~5分）：
        - 是否突出岗位亮点（2分）
        - 是否语言吸引人（2分）
        - 是否符合短标题要求（1分）
        请仅返回一个整数分数（1~5分）。
```

## 配置字段说明

### prompts 字段
| 字段    | 说明                                                   |
| ------- | ------------------------------------------------------ |
| `id`    | 每个提示词的唯一标识符（用于结果记录/引用）            |
| `label` | 提示词在结果中显示的中文/人类可读名                    |
| `raw`   | 实际传递给模型的提示词模板，支持变量（如 `{{input}}`） |

### providers 字段
| 字段 | 说明                                     |
| ---- | ---------------------------------------- |
| `id` | 指定要调用的模型（格式为 `平台:模型名`） |

### tests 字段
| 字段    | 说明                                         |
| ------- | -------------------------------------------- |
| `vars`  | 输入变量定义（对应 prompt 中的 `{{input}}`） |
| `input` | 实际输入内容，会被插入到 prompt 中的位置     |

### eval 字段
| 字段       | 说明                                             |
| ---------- | ------------------------------------------------ |
| `type`     | 自动评分的方式                                   |
| `criteria` | 模型输出中必须包含的关键词                       |
| `rubric`   | LLM 评分的详细标准和要求                         |

## 断言类型

| 类型         | 描述                        |
| ------------ | --------------------------- |
| `equals`     | 严格等于某个期望值          |
| `icontains`  | 忽略大小写包含              |
| `regex`      | 匹配正则表达式              |
| `javascript` | 使用 JS 表达式动态打分      |
| `llm-rubric` | 通过 LLM 自动判分（更智能） |

### JavaScript 打分示例
```javascript
1 / (output.length + 1)
```
这个表达式鼓励模型输出**简洁**的结果：
- 输出越长，分数越低
- 输出越短，分数越高

| 输出内容                    | output.length | 分数 (1 / (len + 1)) |
| --------------------------- | ------------- | -------------------- |
| "Hello"                     | 5             | 1 / 6 ≈ 0.166        |
| "Hello, how are you today?" | 24            | 1 / 25 ≈ 0.04        |

## LLM-Rubric 详解

### 概念
LLM-Rubric 是用一个 LLM 来给另一个 LLM 的输出打分的机制。你需要写一段评分标准，它就能自动判分。

### 配置示例
```yaml
eval:
  automatic:
    - type: llm-rubric
      rubric: |
        评分标准：
        如果输出内容准确、表达清晰且逻辑合理，得5分；
        如果有小问题但大体合理，得3分；
        如果严重偏题或错误，得1分；
        请只输出一个整数分数，不加解释。
      provider: openai:gpt-4
```

### 指定评分模型
```yaml
eval:
  automatic:
    - type: llm-rubric
      rubric: "..."
      provider: openai:gpt-4
```

如果未指定 provider，Promptfoo 默认使用你在 providers 中定义的第一个模型。

## 测试组合原理

Promptfoo 会用所有的 prompt × 所有的 model × 所有的 tests 做**笛卡尔积组合**进行测试。

例如：
- 3 个测试输入
- 2 个提示词
- 2 个模型

总共会产生：3 × 2 × 2 = **12 次测试运行**

## 评估类型

### keyword-match
```yaml
eval:
  automatic:
    - type: keyword-match
      criteria:
        - "总结"
        - "简洁"
```

### regex-match
```yaml
eval:
  automatic:
    - type: regex-match
      pattern: "^\\d+\\.\\s+"
```

### llm-rubric
```yaml
eval:
  automatic:
    - type: llm-rubric
      rubric: |
        请从以下维度为这个输出打分（1~5分）：
        - 准确性（2分）
        - 清晰度（2分）
        - 完整性（1分）
        请仅返回一个整数分数（1~5分）。
```

## 使用技巧

1. **变量命名规范**：使用有意义的变量名，如 `{{input}}`、`{{topic}}` 等
2. **提示词对比**：设计好、坏两种提示词进行对比测试
3. **多模型验证**：使用不同模型验证提示词的通用性
4. **批量测试**：准备多样化的测试用例，覆盖不同场景
5. **自动化评分**：结合多种评分方式，获得更客观的评估结果

## 最佳实践

1. **版本控制**：将 `promptfooconfig.yaml` 纳入版本控制
2. **环境隔离**：使用 `.env` 文件管理 API 密钥
3. **持续集成**：将提示词测试集成到 CI/CD 流程
4. **结果记录**：定期查看和分析测试报告，优化提示词
5. **团队协作**：分享配置文件，建立团队提示词库
