# 前端服务监控

---

## 一、监控的目的

| 目的 | 说明 |
| --- | --- |
| **事前预警** | 在问题影响用户之前提前发现并告警 |
| **事后分析** | 出现故障后快速定位根因、复盘改进 |
| **性能分析** | 量化页面加载与运行时性能，持续优化体验 |
| **业务赋能** | 基于用户行为数据辅助产品决策与业务增长 |

---

## 二、监控的流程

```
数据采集 → 数据上报 → 数据整理和存储 → 数据展示
```

1. **数据采集** — 在前端应用中埋点收集各类监控数据
2. **数据上报** — 将采集到的数据发送到服务端
3. **数据整理和存储** — 服务端对数据进行清洗、聚合与持久化
4. **数据展示** — 通过可视化看板呈现监控信息

## 四、错误监控采集

### 4.1 代码执行错误

| 错误类型 | 说明 | 捕获方式 |
| --- | --- | --- |
| **JS 代码运行错误** | `SyntaxError`、`TypeError`、`ReferenceError` 等 | `window.onerror` / `window.addEventListener('error')` |
| **Promise 异步错误** | 未捕获的 Promise rejection | `window.addEventListener('unhandledrejection')` |

### 4.2 资源加载错误

| 错误类型 | 说明 | 捕获方式 |
| --- | --- | --- |
| **静态资源加载错误** | JS/CSS/图片等资源加载失败 | `window.addEventListener('error', fn, true)`（捕获阶段） |
| **接口请求报错** | HTTP 请求异常（状态码错误、超时等） | 拦截 `XMLHttpRequest` / `fetch` |

### 4.3 跨域脚本报错

- 跨域脚本抛出的错误信息只有 `"Script error."`，无堆栈详情
- 解决方案：为 `<script>` 标签添加 `crossorigin` 属性，同时服务端设置 `Access-Control-Allow-Origin`

### 4.4 框架级错误

| 框架 | 捕获方式 |
| --- | --- |
| **Vue 错误** | `Vue.config.errorHandler` |
| **React 错误** | Error Boundaries（`componentDidCatch`） |

## 五、白屏监控检测

### 5.1 白屏检测难点

- **前端渲染方式的多样性**：客户端渲染（CSR）、服务端渲染（SSR）、静态页面生成（SSG）等，每种模式各不相同，白屏发生的情况也不尽相同
- **白屏原因的不确定**：可能是 JS 执行失败、资源加载超时、接口异常、CSS 未加载等多种原因导致

### 5.2 解决方案

| 方案 | 说明 |
| --- | --- |
| **onerror + DOM 检测** | 监听错误事件，同时检测页面根节点是否有有效 DOM 内容 |
| **Mutation Observer API** | 监听 DOM 变化，判断页面是否在合理时间内完成了渲染 |
| **页面截图检测** | 对页面进行截图，通过像素分析判断是否为空白页面 |
| **elementsFromPoint API 采样对比** | 在页面关键采样点调用 `document.elementsFromPoint()`，检测对应位置是否有 DOM 元素 |
| **饿了么白屏监控方案** | 综合采样检测 + 页面骨架特征比对 |

## 六、性能数据监控采集

### 6.1 Web Vitals 核心指标

| 指标 | 全称 | 说明 |
| --- | --- | --- |
| **FP** | First Paint（白屏时间） | 从页面开始加载到首次有像素绘制的时间 |
| **FCP** | First Contentful Paint（首次内容绘制时间） | 首次有文本、图片等有意义内容绘制的时间 |
| **LCP** | Largest Contentful Paint（最大内容绘制时间） | 视口内最大内容元素渲染完成的时间 |
| **TTFB** | Time to First Byte（第一字节时间） | 从请求发出到收到响应第一个字节的时间 |
| **FID** | First Input Delay（首次输入延迟） | 用户首次与页面交互到浏览器响应的时间 |
| **CLS** | Cumulative Layout Shift（累积布局偏移） | 页面生命周期内非预期布局偏移的累积分数 |

### 6.2 个性化指标

| 指标 | 说明 |
| --- | --- |
| **Long Task（长任务）** | 超过 50ms 的任务，阻塞主线程影响交互响应 |
| **Memory（页面内存）** | 监控页面 JS 堆内存使用情况，预防内存泄漏 |
| **首屏加载时间** | 从页面开始加载到首屏所有内容渲染完成的时间 |

## 七、用户行为数据监控采集

| 数据类型 | 说明 |
| --- | --- |
| **PV / UV** | 页面浏览量 / 独立访客数，衡量流量规模 |
| **页面跳转** | 监听 `hashchange` / `popstate` 记录页面切换路径 |
| **用户点击** | 捕获点击事件，记录用户操作行为 |
| **页面停留时长** | 计算用户在当前页面的停留时间 |
| **页面访问深度** | 记录页面滚动深度，衡量内容消费程度 |

## 八、数据上报

### 8.1 数据上报方式

| 方式 | 说明 | 适用场景 |
| --- | --- | --- |
| **图片打点上报** | 通过 `new Image().src = '上报地址?data=xxx'` 发送请求 | 跨域友好、轻量，适合少量数据 |
| **常规 AJAX 上报** | 使用 `XMLHttpRequest` / `fetch` 发送数据 | 适合大数据量、需要响应确认的场景 |
| **sendBeacon** | `navigator.sendBeacon()` 异步发送数据 | 页面卸载时也能可靠上报，适合离开页面场景 |

### 8.2 数据上报时机

| 时机策略 | 说明 |
| --- | --- |
| **requestIdleCallback / setTimeout 延时上报** | 在浏览器空闲时上报，避免阻塞主线程关键任务 |
| **beforeunload 回调上报** | 在页面即将卸载时集中上报，防止数据丢失 |
| **缓存批量上报** | 缓存上报数据，达到一定数量或一定时间后批量上报，减少请求次数 |

## 九、错误还原

| 手段 | 说明 |
| --- | --- |
| **Source Map 代码还原** | 利用 `.map` 文件将压缩/混淆后的错误堆栈还原为源码位置 |
| **错误行为栈** | 记录用户操作轨迹（点击、跳转、接口调用等），复现错误触发路径 |
| **rrweb 前端录屏** | 使用 rrweb 录制用户操作过程，回放错误发生时的完整页面状态 |

## 十、数据看板

| 模块 | 说明 |
| --- | --- |
| **错误列表** | 展示错误类型、发生次数、影响用户数、错误堆栈等信息 |
| **PV / UV 等** | 展示页面流量趋势、用户访问分布等统计数据 |
| **性能指标** | 展示 FP、FCP、LCP 等核心性能指标的趋势与分位数 |

## 十一、监控报警

### 11.1 报警规则

- 错误率超过阈值触发报警
- 性能指标劣化超过阈值触发报警
- PV/UV 异常波动触发报警
- 自定义规则（如特定接口错误率）

### 11.2 报警方式

- 邮件通知
- 即时通讯工具通知（钉钉 / 飞书 / 企业微信）
- 短信 / 电话告警（严重级别）

---

## 总结

前端服务监控是一套完整的质量保障体系，覆盖了从 **数据采集 → 上报 → 存储 → 展示 → 报警** 的全链路流程。通过错误监控、白屏检测、性能采集、用户行为追踪等手段，结合 Source Map 还原、行为栈录屏等错误定位技术，帮助团队实现：

> **事前预警 → 事后分析 → 性能优化 → 业务赋能**
