# 前端服务监控概述

前端监控的目的很明确，无非就是让我们的产品更完善，更符合我们和用户的需求。运营与产品团队需要关注用户在产品内的行为记录，通过用户的行为记录来优化产品，研发与测试团队则需要关注产品的性能以及异常，确保产品的性能体验以及安全迭代。

而一个完整的前端监控平台至少需要包括三个部分：**数据采集与上报、数据整理和存储、数据展示**。算上需要监控的项目的话，也就是说，至少需要4个项目才能完整的记录前端监控的内容。

## 一些名称的解释

### 监控SDK

通过上面两个现成的框架，大家也大致能看出，我们前端监控到底是要干什么事情

- `页面的性能情况`：包括各阶段加载耗时，一些关键性的用户体验指标等

- `用户的行为情况`：包括PV、UV、访问来路，路由跳转等

- `接口的调用情况`：通过http访问的外部接口的成功率、耗时情况等

- `页面的稳定情况`：各种前端异常等

- `数据上报`：如何将监控捕获到的数据上报

其实完整的监控平台至少分为三大类

- **数据采集与上报**
- 数据整理和存储
- 数据展示

而上面总结的那一大堆，主要就是监控SDK的实现，SDK，其实就是*Software Development Kit*，其实就是提供实现监控的API

### 前端埋点

无论性能，行为还是异常情况，我们都需要在需要监控的项目代码中去监听这些内容。那么具体监听的手段其实就被称之为**前端埋点**。

前端埋点还分为**手动埋点**和**无痕埋点**。

**手动埋点**，就是在要监听的项目中的某段代码或者某个事件中加入一段监听SDK代码，然后对监听的内容进行上报，好处就是可以对关键性行为做出具体的跟踪，坏处是具有侵入性

**无痕埋点**，就是就是对监听的项目进行全部无脑监听，比如点击事件，滚动事件等等，只要触发了就上报。好处就是对代码没有侵入性，坏处当然也很明显无法快速定位关键信息，上报次数多，服务器压力大

## 附录：示例项目代码参考

> 以下代码来自示例项目 `coderstation`（一个问答社区平台），用于辅助理解前端监控在实际项目中的应用场景。该项目包含前端（React）和后端（Express + MongoDB）两部分。

### 附录一：服务端错误处理机制

在实际项目中，**错误监控**首先需要有一套完善的错误分类与处理机制。以下代码展示了后端如何通过自定义错误类来区分不同类型的业务错误，这是后续进行错误数据采集与上报的基础。

```javascript
// coderstation-server/utils/errors.js
// 自定义错误基类与各类业务错误

const { formatResponse } = require("./tools");

/**
 * 业务处理错误基类
 */
class ServiceError extends Error {
  /**
   * @param {*} message 错误消息
   * @param {*} code 错误的消息码
   */
  constructor(message, code) {
    super(message);
    this.code = code;
  }

  // 格式化的返回错误信息
  toResponseJSON() {
    return formatResponse(this.code, this.message, null);
  }
}

// 文件上传错误 (413)
exports.UploadError = class extends ServiceError {
  constructor(message) {
    super(message, 413);
  }
};

// 禁止访问错误 (401)
exports.ForbiddenError = class extends ServiceError {
  constructor(message) {
    super(message, 401);
  }
};

// 验证错误 (406)
exports.ValidationError = class extends ServiceError {
  constructor(message) {
    super(message, 406);
  }
};

// 无资源错误 (406)
exports.NotFoundError = class extends ServiceError {
  constructor() {
    super("not found", 406);
  }
};

// 未知错误 (500)
exports.UnknownError = class extends ServiceError {
  constructor() {
    super("server internal error", 500);
  }
};

module.exports.ServiceError = ServiceError;
```

在 Express 应用的入口文件 `app.js` 中，通过全局错误处理中间件统一捕获并处理所有错误：

```javascript
// coderstation-server/app.js（错误处理部分）

// 自定义错误类的引入
const { ServiceError, UnknownError } = require("./utils/errors");

// 全局错误处理中间件
// 一旦发生了错误，就会到这里来
app.use(function (err, req, res, next) {
  if (err instanceof ServiceError) {
    // 已知的业务错误，返回具体的错误信息
    res.send(err.toResponseJSON());
  } else {
    // 未知错误，返回通用错误信息并抛出（方便日志记录和监控捕获）
    res.send(new UnknownError().toResponseJSON());
    throw err;
  }
});
```

**监控启示**：在实际的监控系统中，我们可以在这类全局错误处理中间件中加入数据上报逻辑，将错误信息（错误类型、错误码、错误堆栈、请求URL、时间戳等）上报到监控平台，从而实现对服务端异常的实时监控。

### 附录二：前端请求拦截器

前端监控中的**接口调用情况监控**，通常需要在请求拦截器中埋点。以下代码展示了项目中 axios 请求拦截器的实现：

```javascript
// coderstation-client/src/api/request.js
import axios from "axios";

const service = axios.create({
  timeout: 5000, // 超时时间
});

// 请求拦截器
service.interceptors.request.use(
  // 正常请求被拦截下来的回调函数
  (config) => {
    // 从本地存储中获取 token
    const token = localStorage.getItem('userToken');
    if(token){
      // 说明本地有 token 信息，在请求头的 Authorization 字段统一添加 token
      config.headers['Authorization'] = "Bearer " + token
    }
    // 放行请求
    return config;
  },
  // 发生错误时的回调函数
  (error) => {
    console.log("请求拦截出错：", error);
  }
);

// 响应拦截器
service.interceptors.response.use(
  // 正常响应被拦截下来时的回调函数
  (response) => {
    const res = response.data;
    if (res.code) {
        switch(res.code){
          case 406:{
            return res;
          }
        }
    } else {
      // 放行响应
      return res;
    }
  },
  // 发生错误时的回调函数
  (error) => {
    console.log("响应拦截出错：", error);
  }
);

export default service;
```

**监控启示**：在请求拦截器中，我们可以记录每次接口请求的URL、请求方法、请求参数、开始时间等信息；在响应拦截器中，记录响应状态码、响应时间、是否成功等数据。这就是**接口调用情况监控**的基本实现思路，也是前端监控SDK的核心功能之一。

### 附录三：前端应用入口

```javascript
// coderstation-client/src/App.js
import React, { useState, useLayoutEffect } from "react";
import { Layout, message } from "antd";
import RouterBefore from "./router/RouteBefore";

import NavHeader from "./components/NavHeader";
import PageFooter from "./components/PageFooter";
import Login from "./components/LoginForm";

import { useDispatch } from "react-redux";
import { changeLoginStatus, initUserInfo } from "./redux/userSlice";
import { getInfo, getUserById } from "./api/user";
import { getTypeList } from "./redux/typeSlice";

import "./css/App.css";

const { Header, Footer, Content } = Layout;

function App() {
  const [isModalOpen, setIsModalOpen] = useState(false);
  const dispatch = useDispatch();

  useLayoutEffect(() => {
    // 第一次进入应用，确认用户的登录状态
    async function whoami() {
      const result = await getInfo();
      if (result.data) {
        // token 验证成功
        const { data } = await getUserById(result.data._id);
        dispatch(initUserInfo(data));
        dispatch(changeLoginStatus(true));
      } else {
        message.warning(result.msg);
        localStorage.removeItem("userToken");
      }
    }
    if (localStorage.getItem("userToken")) {
      whoami();
    }
    // 填充仓库的类型列表
    dispatch(getTypeList());
  }, []);

  // ... 渲染逻辑省略
}

export default App;
```

**监控启示**：在应用入口处，正是初始化监控SDK的最佳位置（类似于 Sentry 在 `Sentry.init()` 中的做法）。可以在此处配置监控的采样率、上报地址、用户信息等，实现对整个应用的全方位监控覆盖。
