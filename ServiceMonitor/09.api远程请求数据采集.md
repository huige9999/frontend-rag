# API 远程请求数据采集

---

## 一、API 请求监控概述

前端项目中，页面与后端的数据交互几乎全部通过 HTTP 请求完成（AJAX）。API 请求的质量直接影响用户体验：

| 关注点 | 说明 |
| --- | --- |
| **请求耗时** | 接口响应过慢会导致页面卡顿、白屏 |
| **请求成功率** | 接口报错可能导致功能不可用、数据丢失 |
| **请求频率** | 过多的请求可能意味着接口设计不合理或存在重复请求 |
| **错误定位** | 出现故障时需要快速定位是哪个接口、什么原因导致的 |

目前前端发起 HTTP 请求主要有两种方式：

| 方式 | 全局对象 | 典型场景 |
| --- | --- | --- |
| **XMLHttpRequest** | `XMLHttpRequest` | axios（浏览器端）、传统 jQuery.ajax 等 |
| **Fetch API** | `window.fetch` | 原生 fetch 调用、部分现代库 |

因此，要实现 API 请求的全面监控，就需要**同时拦截 XMLHttpRequest 和 Fetch** 两种请求方式。我们的监控 SDK 通过"猴子补丁"（Monkey Patch）的方式，在不修改业务代码的前提下，重写这两个原生 API 的关键方法，从而捕获所有请求的数据。

### 整体架构

```
┌─────────────────────────────────────────────────────┐
│                  SDK 入口 (index.js)                 │
│                  monitor.init(options)               │
└──────────────────────┬──────────────────────────────┘
                       │ 调用 api()
┌──────────────────────▼──────────────────────────────┐
│              API 监控入口 (api/index.js)              │
│              统一调度 XHR 和 Fetch 拦截               │
└────────┬───────────────────────────┬────────────────┘
         │                           │
┌────────▼────────┐       ┌─────────▼────────┐
│  XHR 拦截        │       │  Fetch 拦截       │
│  (api/xhr.js)   │       │  (api/fetch.js)  │
│  重写 open/send  │       │  重写 window.fetch│
└────────┬────────┘       └─────────┬────────┘
         │                           │
         └───────────┬───────────────┘
                     │ lazyReportCache('api', data)
┌────────────────────▼────────────────────────────────┐
│              数据上报 (report/index.js)               │
│              缓存合并 → sendBeacon 上报               │
└─────────────────────────────────────────────────────┘
```

## 二、XHR 拦截监控 (xhr.js)

### 2.1 XMLHttpRequest 基础回顾

在使用原生 `XMLHttpRequest` 发送请求时，核心流程如下：

```javascript
const xhr = new XMLHttpRequest();
xhr.open('GET', '/api/user');  // 第1步：初始化请求，指定方法和URL
xhr.send();                     // 第2步：发送请求
xhr.onloadend = function() {    // 第3步：请求结束回调
  console.log(xhr.status);      // 获取响应状态码
};
```

关键方法：
- **`open(method, url, ...)`** — 初始化请求，传入请求方法和 URL
- **`send(body)`** — 发送请求

要拦截 XHR 请求，我们就需要重写这两个方法。

### 2.2 重写思路

1. 保存原始的 `open` 和 `send` 方法的引用
2. 用自定义函数替换原型上的 `open` 和 `send`
3. 在自定义函数中添加数据采集逻辑，最后仍调用原始方法

### 2.3 完整代码解析

```javascript
// api/xhr.js
import { lazyReportCache } from '../report';

// 保存 XMLHttpRequest 原型上的原始方法引用
const originalPrototype = XMLHttpRequest.prototype;
const originalOpen = originalPrototype.open;
const originalSend = originalPrototype.send;

function overWriteOpenAndSend() {
  // 重写 open 方法 —— 捕获请求方法和URL
  originalPrototype.open = function newOpen(...args) {
    // args[0] = method, args[1] = url
    this.url = args[1];
    this.method = args[0];
    originalOpen.apply(this, args);
  };

  // 重写 send 方法 —— 捕获请求耗时和响应状态
  originalPrototype.send = function newSend(...args) {
    this.startTime = Date.now(); // 记录请求开始时间

    const onLoadEnd = () => {
      this.endTime = Date.now();
      this.duration = this.endTime - this.startTime; // 计算请求耗时

      const { status, duration, startTime, endTime, url, method } = this;

      const reportData = {
        status,                                          // HTTP 状态码
        duration,                                        // 请求耗时（ms）
        startTime,                                       // 开始时间戳
        endTime,                                         // 结束时间戳
        url,                                             // 请求地址
        method: method || 'GET',                         // 请求方法，默认 GET
        success: status >= 200 && status < 300,          // 是否成功（2xx）
        subType: 'xhr',                                  // 标记请求类型为 xhr
      };

      lazyReportCache('api', reportData); // 延迟批量上报

      this.removeEventListener('loadend', onLoadEnd, true);
    };

    // loadend 事件：请求结束时触发（无论成功、失败还是中止）
    this.addEventListener('loadend', onLoadEnd, true);
    originalSend.apply(this, args);
  };
}

export default function xhr() {
  overWriteOpenAndSend();
}
```

### 2.4 关键技术点

**1. 在 `open` 中保存请求信息到 `this`**

```javascript
originalPrototype.open = function newOpen(...args) {
  this.url = args[1];
  this.method = args[0];
  originalOpen.apply(this, args);
};
```

因为 `open` 被调用时 xhr 实例已经存在，我们可以将 `url` 和 `method` 挂载到实例（`this`）上，这样在后续的 `send` 方法中就能通过 `this.url` 和 `this.method` 读取到这些信息。

**2. 在 `send` 中记录开始时间并监听 `loadend`**

```javascript
this.startTime = Date.now();
this.addEventListener('loadend', onLoadEnd, true);
```

- `send` 是请求实际发出的时刻，因此在这里记录开始时间最为准确
- `loadend` 事件在请求结束时触发（无论成功 `load`、失败 `error` 还是中止 `abort`），是最合适的结束时机

**3. 成功判断逻辑**

```javascript
success: status >= 200 && status < 300
```

HTTP 状态码在 200-299 范围内表示请求成功，这是 HTTP 规范的定义。

**4. 原型级别重写，一次覆盖所有实例**

我们重写的是 `XMLHttpRequest.prototype` 上的方法，这意味着之后所有通过 `new XMLHttpRequest()` 创建的实例都会使用我们重写后的方法，无需修改任何业务代码。

## 三、Fetch 拦截监控 (fetch.js)

### 3.1 Fetch API 基础回顾

```javascript
fetch('/api/user', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ name: 'test' }),
})
  .then(res => {
    console.log(res.status); // 状态码
    console.log(res.ok);     // 是否成功（布尔值）
    return res.json();
  })
  .catch(err => {
    console.error('请求失败', err);
  });
```

与 XHR 不同，`fetch` 是一个全局函数（`window.fetch`），返回 Promise。因此拦截方式也有所不同——直接替换 `window.fetch`。

### 3.2 重写思路

1. 保存原始 `window.fetch` 的引用
2. 用自定义函数替换 `window.fetch`
3. 在自定义函数中记录开始时间，利用 Promise 的 `.then()` 和 `.catch()` 捕获响应和错误

### 3.3 完整代码解析

```javascript
// api/fetch.js
import { lazyReportCache } from '../report';

const originalFetch = window.fetch;

function overWriteFetch() {
  window.fetch = function newFetch(url, config) {
    const startTime = Date.now();

    const reportData = {
      startTime,
      url,
      method: config?.method || 'GET',  // 从 config 中读取方法，默认 GET
      subType: 'fetch',                  // 标记请求类型为 fetch
    };

    return originalFetch(url, config)
      .then(res => {
        reportData.endTime = Date.now();
        reportData.duration = reportData.endTime - reportData.startTime;
        reportData.status = res.status;
        reportData.success = res.ok; // res.ok 是 fetch API 提供的布尔值

        lazyReportCache('api', reportData);

        return res; // 返回原始响应，不影响业务逻辑
      })
      .catch(err => {
        reportData.endTime = Date.now();
        reportData.duration = reportData.endTime - reportData.startTime;
        reportData.status = 0;        // 网络错误时状态码设为 0
        reportData.success = false;

        lazyReportCache('api', reportData);

        return err; // 返回错误，不影响业务逻辑
      });
  };
}

export default function fetch() {
  overWriteFetch();
}
```

### 3.4 关键技术点

**1. 替换 `window.fetch`**

```javascript
const originalFetch = window.fetch;
window.fetch = function newFetch(url, config) { ... };
```

直接在全局对象上替换 `fetch` 函数，所有后续的 `fetch()` 调用都会走到我们的包装函数。

**2. 使用 Promise 链捕获响应和错误**

```javascript
return originalFetch(url, config)
  .then(res => { /* 成功处理 */ })
  .catch(err => { /* 错误处理 */ });
```

- `.then()` 中获取响应对象 `res`，可以读取 `res.status`（状态码）和 `res.ok`（是否成功）
- `.catch()` 捕获网络错误等异常情况，此时没有响应对象，状态码设为 `0`

**3. 透传返回值，不影响业务**

```javascript
// then 中
return res;
// catch 中
return err;
```

重写后的 `fetch` 仍然返回原始的响应（或错误），业务代码的 `.then()` / `.catch()` 链不受影响。

**4. Fetch 中的 `res.ok` vs XHR 中的状态码判断**

| 方式 | 成功判断 |
| --- | --- |
| XHR | `status >= 200 && status < 300` 手动判断 |
| Fetch | `res.ok` — Fetch API 原生提供的布尔值属性 |

两种方式本质相同，都是判断状态码是否在 2xx 范围内。

### 3.5 XHR 与 Fetch 拦截对比

| 对比项 | XHR 拦截 | Fetch 拦截 |
| --- | --- | --- |
| **拦截目标** | `XMLHttpRequest.prototype.open/send` | `window.fetch` |
| **拦截方式** | 原型方法重写 | 全局函数替换 |
| **时机捕获** | `loadend` 事件 | Promise 的 `then/catch` |
| **成功判断** | `status >= 200 && status < 300` | `res.ok` |
| **错误处理** | `loadend` 统一处理（含失败） | `.catch()` 单独处理 |
| **网络错误状态码** | `status` 为 0 | 手动设为 `0` |

## 四、API 监控入口 (api/index.js)

### 4.1 统一调度

```javascript
// api/index.js
import xhr from './xhr';
import fetch from './fetch';

export default function api() {
  xhr();
  fetch();
}
```

这个模块作为 API 监控的统一入口，职责非常清晰：

1. **导入** XHR 和 Fetch 两个拦截模块
2. **统一调用** — 外部只需调用 `api()`，即可同时启用两种请求方式的监控

### 4.2 设计思路

采用这种入口模式的优点：

| 优点 | 说明 |
| --- | --- |
| **简化调用** | SDK 入口只需调用一个 `api()` 函数 |
| **模块解耦** | XHR 和 Fetch 各自独立，互不影响 |
| **可扩展** | 未来如果需要监控其他请求方式（如 WebSocket），只需在此添加即可 |
| **可独立禁用** | 如果只需要监控其中一种，可以注释掉另一个 |

## 五、SDK 集成 (index.js)

### 5.1 在 SDK 中注册 API 监控

```javascript
// index.js
import { setConfig } from "./config";
import performance from "./performance";
import error from "./error";
import { tracker, autoTracker } from "./action";
import { pv, pageStayTime, pageChange, onVueRouter } from "./behavior";
import api from "./api";  // 引入 API 监控模块

const monitor = {
  init(options = {}) {
    console.log("init");
    setConfig(options);   // 配置全局参数
    error();              // 错误监听处理
    performance();        // 性能监听处理
    autoTracker();        // 自动埋点
    pv();                 // page view
    api();                // API 远程请求数据采集
  },
  tracker,
  pageStayTime,
  pageChange,
  onVueRouter,
};

export default monitor;
```

### 5.2 集成要点

**1. 初始化顺序**

```javascript
init(options = {}) {
  setConfig(options);  // 1. 先配置全局参数（上报地址等）
  error();             // 2. 错误监听
  performance();       // 3. 性能监听
  autoTracker();       // 4. 自动埋点
  pv();                // 5. PV 统计
  api();               // 6. API 监控
}
```

`setConfig` 必须最先执行，因为后续所有模块在上报数据时都需要读取配置（如 `reportUrl`）。`api()` 放在较后位置执行，因为它会立即重写原生 API，需要确保在此之前不会遗漏任何请求。

**2. 模块化设计**

每个监控功能都是独立的模块，通过 `init()` 统一注册。这样设计的好处是：
- 各模块职责单一，便于维护和测试
- 可以灵活地开启或关闭某个监控功能
- 新增监控类型时只需新建模块并在 `init` 中注册

### 5.3 使用方式

在业务项目的入口文件中：

```javascript
import monitor from 'monitor-sdk';

monitor.init({
  appId: 'my-project',
  userId: 'user-123',
  reportUrl: 'https://monitor.example.com/report',
});
```

初始化后，SDK 会自动拦截所有 XHR 和 Fetch 请求，并将采集到的数据上报到指定地址。

## 六、配置更新 (config/index.js)

### 6.1 配置结构

```javascript
// config/index.js
const config = {
  appId: 'monitor-sdk-demo',   // 项目名称
  userId: 'ys',                 // 用户标识
  reportUrl: 'http://localhost:3001/report/actions', // 上报地址
  trackerAll: false,            // 是否全埋点
  vue: {
    Vue: null,
    router: null,
  },
  ua: navigator.userAgent,     // 用户浏览器和系统信息
};

export default config;

export function setConfig(options) {
  for (const key in config) {
    if (options[key]) {
      config[key] = options[key];
    }
  }
}
```

### 6.2 配置参数说明

| 参数 | 类型 | 默认值 | 说明 |
| --- | --- | --- | --- |
| `appId` | string | `'monitor-sdk-demo'` | 应用标识，用于区分不同项目 |
| `userId` | string | `'ys'` | 用户标识，用于关联用户数据 |
| `reportUrl` | string | `'http://localhost:3001/report/actions'` | 数据上报的服务端地址 |
| `trackerAll` | boolean | `false` | 是否开启全埋点模式 |
| `vue` | object | `{ Vue: null, router: null }` | Vue 相关配置（用于路由监控） |
| `ua` | string | `navigator.userAgent` | 用户浏览器信息（自动获取） |

### 6.3 setConfig 合并逻辑

```javascript
export function setConfig(options) {
  for (const key in config) {
    if (options[key]) {
      config[key] = options[key];
    }
  }
}
```

采用**浅合并**策略：遍历用户传入的 `options`，如果某个 key 在默认配置 `config` 中存在且用户提供了有效值，就用用户的值覆盖默认值。这种方式简单直接，适合当前配置结构。

## 七、数据上报流程

API 监控采集到的数据通过 `lazyReportCache` 进行上报，整个流程如下：

```
请求发出 → 采集数据 → lazyReportCache() → 加入缓存 → 等待 3 秒
                                                       ↓
                                            取出缓存中所有同类数据
                                                       ↓
                                            report('api', [data1, data2, ...])
                                                       ↓
                                            requestIdleCallback / setTimeout
                                                       ↓
                                            navigator.sendBeacon() 上报
```

### 7.1 上报数据格式

每条 API 监控数据包含以下字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `status` | number | HTTP 状态码（网络错误时为 0） |
| `duration` | number | 请求耗时（毫秒） |
| `startTime` | number | 请求开始时间戳 |
| `endTime` | number | 请求结束时间戳 |
| `url` | string | 请求地址 |
| `method` | string | 请求方法（GET / POST / PUT / DELETE 等） |
| `success` | boolean | 请求是否成功 |
| `subType` | string | 请求子类型（`'xhr'` 或 `'fetch'`） |

### 7.2 延迟批量上报的优势

```javascript
// report/index.js 中的 lazyReportCache
let timer = null;
export function lazyReportCache(type, data, timeout = 3000) {
  addCache(type, data);  // 先缓存
  clearTimeout(timer);
  timer = setTimeout(() => {
    const dataMap = getCache();
    if (dataMap.size) {
      for (const [type, data] of dataMap) {
        report(type, data);  // 批量上报
      }
      clearCache();
    }
  }, timeout);
}
```

| 优势 | 说明 |
| --- | --- |
| **减少请求次数** | 多次采集的数据合并为一次上报，减轻服务器压力 |
| **避免阻塞** | 通过 `requestIdleCallback` 在浏览器空闲时上报，不影响用户操作 |
| **防抖机制** | 每次新数据到来时重置定时器，确保在数据"平静期"后统一上报 |

## 八、总结

### 核心知识点

1. **Monkey Patch（猴子补丁）** — 通过重写原生 API 的方法来实现无侵入式拦截，这是前端监控 SDK 的核心技术手段
2. **双通道拦截** — 同时拦截 XMLHttpRequest 和 Fetch API，确保覆盖所有 HTTP 请求
3. **原型重写 vs 全局替换** — XHR 通过重写原型方法拦截，Fetch 通过替换全局函数拦截，两种方式各有适用场景
4. **延迟批量上报** — 通过缓存 + 定时器的方式合并上报数据，减少网络请求，提升性能
5. **模块化架构** — API 监控作为独立模块，与错误监控、性能监控等模块并行，通过 SDK 入口统一调度

### 文件结构总览

```
src/
├── api/
│   ├── index.js      # API 监控入口，统一调度 XHR 和 Fetch 拦截
│   ├── xhr.js        # XMLHttpRequest 拦截，重写 open/send 方法
│   └── fetch.js      # Fetch API 拦截，重写 window.fetch
├── config/
│   └── index.js      # 全局配置与配置合并
├── report/
│   └── index.js      # 数据上报（缓存 + sendBeacon）
└── index.js          # SDK 入口，注册所有监控模块
```
