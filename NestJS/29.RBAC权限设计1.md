# 29. RBAC权限设计1

## 什么是 RBAC 模型

一个用户在使用系统做某些操作的时候，系统会去数据库或者其他持久化的地方查询该用户所拥有的权限，然后根据查询出的结果判断此次操作是否正常。

简单的情况，一张表存储用户的权限，然后直接查询判断即可。但**用户量足够大又或者权限非常多**的话怎么办呢？一个新的系统需要接入用户、权限的时候，又该怎么办？

为了解决前述的问题，我们将引入 **RBAC** 权限管理设计。

> **RBAC（Role-Based Access Control）** 的三要素即**用户**、**角色**与**权限**。 用户通过赋予的角色，执行角色所拥有的权限。

RBAC 引入之后的流程：用户在进入系统之后，会先进行角色判断，再根据对应的角色查询所匹配的权限，最后根据返回结果来判断是否可执行。

这种记录每个用户有什么权限的方式，叫做**ACL**(`Access Control List`)访问控制表。而 RBAC 是先给**角色分配权限**，然后给**用户分配对应的角色**。

对比两种方案：

| 方案                     | 用户量       | 权限数 | 权限表数据量       |
| ------------------------ | ------------ | ------ | ------------------ |
| **用户 -> 权限**         | **1**        | **10** | **10 * 1**         |
| **用户 -> 权限**         | **100,000**  | **10** | **10 * 100,000**   |
| **用户 -> 角色 -> 权限** | **1**        | **10** | **10 * Role**      |
| **用户 -> 角色 -> 权限** | **100,000**  | **10** | **10 * Role**      |

使用 RBAC 后，当用户进入系统，先查询用户对应的角色，再查询角色映射对应的权限表，即便是一个角色对应一个用户，查询量也就是在 **10 * 10**，比直接查询百万数据表的数据量直线下降。

> RBAC 模型还有细分：RBAC0（最简单）、RBAC1（角色继承）、RBAC2（职责分离）、RBAC3。本课使用 **RBAC0**。

### RBAC 表结构设计

用户、角色、权限都是**多对多关系**：
- 用户 <-> 角色（中间表 `user_role_relation`）
- 角色 <-> 权限（中间表 `role_permission_relation`）

## 关键代码

### 1. 实体定义（Entity）

**User 实体** - 用户与角色多对多关系：

```typescript
@Entity()
export class User {
  @PrimaryGeneratedColumn()
  id: number;

  @Column()
  username: string;

  @Column()
  password: string;

  @ManyToMany(() => Role)
  @JoinTable({ name: 'user_role_relation' })
  roles: Role[];
}
```

**Role 实体** - 角色与权限多对多关系：

```typescript
@Entity()
export class Role {
  @PrimaryGeneratedColumn()
  id: number;

  @Column()
  name: string;

  @ManyToMany(() => Permission)
  @JoinTable({ name: 'role_permission_relation' })
  permissions: Permission[];
}
```

**Permission 实体**：

```typescript
@Entity()
export class Permission {
  @PrimaryGeneratedColumn()
  id: number;

  @Column()
  name: string;
}
```

### 4. 登录与 JWT 签发

登录时通过 `relations` 关联查询角色，签发 JWT 携带用户名和角色信息：

```typescript
// UserService
async login(loginUserDto: LoginUserDto) {
  const user = await this.entityManager.findOne(User, {
    where: { username: loginUserDto.username },
    relations: { roles: true },
  });
  if (!user) throw new HttpException('用户不存在', HttpStatus.ACCEPTED);
  if (user.password !== loginUserDto.password)
    throw new HttpException('密码错误', HttpStatus.ACCEPTED);
  return user;
}

// UserController
@Post('login')
async login(@Body(ValidationPipe) user: LoginUserDto) {
  const result = await this.userService.login(user);
  const token = this.jwtService.sign({
    user: { username: result.username, roles: result.roles },
  });
  return { token };
}
```

### 5. 全局登录守卫（LoginGuard）

使用 `Reflector` 读取自定义装饰器 `@LoginRequired`，仅对标记的路由要求登录验证：

```typescript
@Injectable()
export class LoginGuard implements CanActivate {
  @Inject() private reflector: Reflector;
  @Inject(JwtService) private jwtService: JwtService;

  canActivate(context: ExecutionContext) {
    const request = context.switchToHttp().getRequest();

    // 未标记 @LoginRequired 的路由直接放行
    const isLoginRequired = this.reflector.getAllAndOverride('LoginRequired', [
      context.getClass(),
      context.getHandler(),
    ]);
    if (!isLoginRequired) return true;

    const authorization = request.headers.authorization;
    if (!authorization) throw new UnauthorizedException('请先登录');

    try {
      const token = authorization.split(' ')[1];
      const data = this.jwtService.verify(token);
      (request as any).user = data.user; // 将用户信息挂到 request 上
      return true;
    } catch (e) {
      throw new UnauthorizedException('token 失效，请重新登录');
    }
  }
}
```

### 6. 全局权限守卫（PermissionGuard）

从 request 获取用户角色，查询对应权限，与路由装饰器 `@PermissionRequired` 声明的所需权限进行匹配：

```typescript
@Injectable()
export class PermissionGuard implements CanActivate {
  @Inject(UserService) private userService: UserService;
  @Inject() private reflector: Reflector;

  async canActivate(context: ExecutionContext) {
    const request = context.switchToHttp().getRequest();
    const user = request.user;
    if (!user) return true;

    // 查询用户所有角色的权限
    const roles = await this.userService.findRolesByIds(
      user.roles.map((item) => item.id),
    );
    const permissions = roles.reduce((acc, cur) => {
      acc.push(...cur.permissions);
      return acc;
    }, []);

    // 获取路由所需权限
    const permissionRequired = this.reflector.getAllAndOverride(
      'PermissionRequired',
      [context.getClass(), context.getHandler()],
    );
    if (!permissionRequired) return true;

    // 逐个匹配权限
    for (const curPermission of permissionRequired) {
      const found = permissions.find((item) => item.name === curPermission);
      if (!found) throw new UnauthorizedException('该接口你没有访问的权限');
    }
    return true;
  }
}
```

### 7. 自定义装饰器

通过 `SetMetadata` 定义路由元数据，配合守卫的 `Reflector` 读取：

```typescript
import { SetMetadata } from '@nestjs/common';

export const LoginRequired = () => SetMetadata('LoginRequired', true);

export const PermissionRequired = (...permissions: string[]) =>
  SetMetadata('PermissionRequired', permissions);
```

### 8. 全局守卫注册与 UserService 导出

```typescript

  AppService,
  { provide: 'APP_GUARD', useClass: LoginGuard },
  { provide: 'APP_GUARD', useClass: PermissionGuard },
],

  controllers: [UserController],
  providers: [UserService],
  exports: [UserService],
```


## 核心流程总结

1. **User/Role/Permission 三个实体通过多对多关系建立 RBAC0 模型**
2. **登录后签发 JWT，携带用户名和角色信息**
3. **全局 LoginGuard 拦截请求**，通过 `@LoginRequired` 装饰器标记需要登录的路由
4. **全局 PermissionGuard 检查权限**，通过 `@PermissionRequired` 装饰器声明路由所需权限，与用户角色权限做匹配
5. 两个守卫通过 `APP_GUARD` 全局注册，无需在每个 Controller 手动添加
