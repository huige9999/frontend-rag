# 请求库的封装

## 课程概述

本节课讲解如何从零到一封装一套适配公司业务的前端请求库。课程从背景分析出发，阐述了为什么现有公共请求库无法满足实际需求，然后深入讲解了请求库的三层架构设计、依赖倒置原则的应用、请求缓存与幂等的实现、以及样板代码的自动化生成与补丁机制。

**核心知识点：**

- 请求库三层架构设计（请求实现层、request-core、request-bus）
- DIP（依赖倒置原则）在请求库中的应用
- 请求缓存机制的实现
- 请求幂等性保障
- 基于接口平台的样板代码自动生成
- 补丁机制覆盖样板代码

## 一、背景：为什么需要自行封装请求库

### 1.1 现有方案的不足

**axios 的问题**

axios 虽然成熟，但只是一个基础库，没有提供上层功能：

1. 请求重试
2. 请求缓存
3. 请求幂等
4. 请求串行
5. 请求并发

**VueRequest / SWR 的问题**

1. 与上层框架过度绑定，开发场景受限，无法提供统一的 API
2. 成熟度不够，issue 回复不及时，存在风险
3. 没有聚合基础请求库，仍然需要手动整合

**最关键的问题**

公共库不包含公司内部制定的协议规范，即便使用公共库，也必须针对它们做二次封装。

### 1.2 结论

需要自行封装一套适配公司业务的前端请求库。

## 二、库结构的宏观设计

### 2.1 三层架构

整个库结构包含三层，从下往上依次是：

- **请求实现层（request-imp）**：提供请求基本功能
- **request-core**：提供网络上层控制，比如请求串行、请求并行、请求重试、请求防重等功能
- **request-bus**：为请求绑定业务功能，该层接入公司内部协议规范和接口文档，向外提供业务接口 API

### 2.2 优化设计 —— 应用 DIP 原则

在三层中，请求实现层的实现有多种方式（XHR / fetch / axios），这种多样性会导致基础层不稳定，而不稳定性会传导到上层。

**解决方案：依赖倒置原则（DIP）**

基于 DIP（Dependence Inversion Principle，依赖倒置原则），彻底将 `request-core` 和请求的实现解耦，TypeScript 的类型系统让这一切的落地成为了可能。

#### request-core 代码示意

`request-core` 定义接口但不负责实现，通过依赖注入的方式获取具体的请求实例：

```typescript
// 定义接口，不负责实现
export interface Requestor {
  get(url: string, options: RequestOptions): Promise<Response>;
  // 略...
}

// 本模块的大部分功能都需要使用到 requestor
let req: Requestor;

export function inject(requestor: Requestor) {
  req = requestor;
}

export function useRequestor() {
  return req;
}

// 创建一个可以重试的请求
export function createRetryRequestor(maxCount = 5) {
  const req = useRequestor();
  // 进一步配置 req
  return req;
}

// 创建一个并发请求
export function createParallelRequestor(maxCount = 4) {
  const req = useRequestor();
  // 进一步配置 req
  return req;
}
```

**代码解读：**

- `Requestor` 接口定义了请求的抽象契约（如 `get` 方法），不包含任何具体实现
- `inject` 函数实现依赖注入（DI），由外部注入具体的请求实现
- `useRequestor` 用于获取当前注入的请求实例
- `createRetryRequestor` 和 `createParallelRequestor` 等工厂函数基于注入的实例创建增强版请求器

#### request-axios-imp 代码示意

`request-axios-imp` 是基于 axios 的具体实现：

```typescript
import { Requestor } from 'request-core';
import axios from 'axios';

const ins = axios.create();

export const requestor: Requestor = {
  get(url, options?) {
    // 使用 axios 实现
  },
  // 其他请求方法
};
```

**代码解读：**

- 实现了 `Requestor` 接口，使用 axios 作为底层请求引擎
- 通过 `axios.create()` 创建独立的 axios 实例，避免全局污染

#### request-bus 代码示意

`request-bus` 负责注入具体实现并对外暴露业务 API：

```typescript
// 为 request-core 注入 requestor 的具体实现
import { inject } from 'request-core';
import { requestor } from 'request-axios-imp';
inject(requestor);
```

**代码解读：**

- `request-bus` 作为最上层，在初始化时将 axios 实现注入到 `request-core`
- 当实现变动时（如从 axios 切换到 fetch），仅需修改注入来源即可

#### 切换实现示例

新增 `request-fetch-imp` 后，仅需修改依赖来源：

```typescript
import { Requestor } from 'request-core';

export const requestor: Requestor = {
  get(url, options?) {
    // 使用 fetch 实现
  },
  post(url, data, options?) {
    // 使用 fetch 实现
  },
  // 其他请求方法
};
```

```diff
- import { requestor } from 'request-axios-imp';
+ import { requestor } from 'request-fetch-imp';
inject(requestor);
```

## 三、请求缓存

### 3.1 功能说明

请求缓存是指创建一个带有缓存的请求，当没有命中缓存时发送请求并缓存结果，当有缓存时直接返回缓存。

```javascript
const req = createCacheRequestor();
req.get('/a'); // 发送请求
req.get('/a'); // 使用缓存
req.get('/b'); // 发送请求
req.get('/b'); // 使用缓存
```

### 3.2 核心设计问题

**问题一：请求结果怎么存？存在哪？缓存键是什么？**

用户可以指定缓存方案（内存/持久化）和缓存键：

```javascript
const req = createCacheRequestor({
  key: (config) => {
    // config 为某次请求的配置
    return config.pathname; // 使用 pathname 作为缓存键
  },
  persist: true, // 是否开启持久化缓存
});
```

为了抹平不同存储方案的差异，设计统一的存储接口：

```typescript
export interface CacheStore {
  has(key: string): Promise<boolean>;
  set<T>(key: string, ...values: T[]): Promise<void>;
  get<T>(key: string): Promise<T>;
  // 其他字段
}

export function useCacheStore(isPersist: boolean): CacheStore {
  if (!isPersist) {
    return createMemoryStore();
  } else {
    return createStorageStore();
  }
}
```

**代码解读：**

- `CacheStore` 接口定义了统一的缓存操作契约（`has`、`set`、`get`）
- `useCacheStore` 根据是否持久化返回不同的存储实现（内存或 Storage）
- 通过接口隔离，存储方案的变动不会影响上层缓存逻辑

**问题二：缓存何时失效？**

支持基于时间和自定义条件两种失效策略：

```javascript
const req = createCacheRequestor({
  duration: 1000 * 60 * 60, // 缓存时间，单位毫秒
  isValid(key, config) {
    // 自定义缓存是否有效，提供该配置后 duration 配置失效
    // key 表示缓存键，config 表示此次请求配置
    // 返回 true 表示缓存有效，返回 false 缓存无效
  },
});
```

### 3.3 实现核心逻辑

```javascript
function createCacheRequestor(cacheOptions) {
  const options = normalizeOptions(cacheOptions); // 参数归一化
  const store = useCacheStore(options.persist); // 使用缓存仓库
  const req = useRequestor(); // 获得请求实例
  // 对请求进行配置（见下）
  return req;
}
```

**请求前拦截 —— 命中缓存则直接返回：**

```javascript
// 注册请求发送前的事件
req.on('beforeRequest', async (config) => {
  const key = options.key(config); // 获得缓存键
  const hasKey = await store.has(key); // 是否存在缓存
  if (hasKey && options.isValid(key, config)) {
    // 存在缓存并且缓存有效，返回缓存结果
  }
});
```

**请求后拦截 —— 缓存响应结果：**

```javascript
req.on('responseBody', (config, resp) => {
  const key = options.key(config); // 获得缓存键
  store.set(key, resp.toPlain());
});
```

**代码解读：**

- 通过事件机制（`beforeRequest`、`responseBody`）实现缓存逻辑的注入
- 请求前检查缓存，命中则短路请求；请求后将结果写入缓存
- `normalizeOptions` 确保配置项具有合理的默认值

## 四、请求幂等

### 4.1 幂等性概念

幂等性是一个数学概念：`f(n) = 1^n`，无论 n 的值是多少，f(n) 始终为 1。

在网络请求中，很多接口都要求幂等性，比如支付：同一订单多次支付和一次支付对用户余额的影响应该是一样的。

### 4.2 核心思路

保证幂等的关键：**要求幂等的请求不能重复提交**。

**如何定义"重复"？** 请求方法、请求头、请求体完全一致即为重复。

使用 MD5 哈希将它们编码成一个唯一字符串：

```javascript
function hashRequest(req) {
  const spark = new SparkMD5();
  spark.append(req.url);
  for (const [key, value] of req.headers) {
    spark.append(key);
    spark.append(value);
  }
  spark.append(req.body);
  return spark.end();
}
```

**代码解读：**

- 使用 `SparkMD5` 库生成请求特征的哈希值
- 将 URL、请求头、请求体拼接后计算 MD5，确保相同的请求产生相同的哈希值
- 哈希值作为缓存键，用于判断是否为重复请求

### 4.3 实现方式

幂等请求可以直接基于缓存功能实现：

```javascript
function createIdempotentRequestor(genKey) {
  return createCacheRequestor({
    key: (config) => (genKey ? genKey(config) : hashRequest(config)),
    persist: false,
  });
}
```

**代码解读：**

- `createIdempotentRequestor` 复用了 `createCacheRequestor` 的缓存能力
- 默认使用 `hashRequest` 生成缓存键，也支持自定义键生成函数
- `persist: false` 表示幂等缓存仅存于内存，不需要持久化

## 五、样板代码的自动生成

### 5.1 背景

公司的 API 接口数量庞大且时常变化，如果 `request-bus` 层全部人工处理不仅耗时，而且容易出错。因此通过标准化工具实现自动化。

### 5.2 接口平台

公司自研接口平台，制定接口文档时可以设置对应字段（如是否需要认证、是否幂等、是否缓存等）。

接口平台输出的标准 JSON 描述文件示例：

```json
{
  "endpoints": {
    "article": {
      "publishArticle": {
        "path": "/api/article",
        "description": "发布文章",
        "method": "POST",
        "auth": true,
        "idempotent": true,
        "cache": false,
        "pager": false
      },
      "getArticles": {
        "path": "/api/article",
        "description": "获取文章",
        "method": "GET",
        "auth": false,
        "idempotent": false,
        "cache": false,
        "pager": true
      }
    }
  }
}
```

### 5.3 样板代码生成工具

使用 Node.js 开发的命令行工具 `request-templet-cli`，根据 JSON 描述文件自动生成样板代码。

生成的样板代码示例：

```typescript
// request-bus/templet/article.ts

import { createIdempotentRequest } from 'request-core';

/**
 * 发布文章
 */
export const publishArticle = (() => {
  const req = createIdempotentRequest();
  return async (article: Article) => {
    return req.post('/api/article', article).then((resp) => resp.json());
  };
})();

/**
 * 获取文章
 */
export const getArticles = (() => {
  return async (page: number, size: number) => {
    return req
      .get('/api/article', {
        params: {
          page,
          size,
        },
      })
      .then((resp) => resp.json());
  };
})();
```

**代码解读：**

- 使用 IIFE（立即执行函数表达式）模式导出函数，在闭包中初始化请求实例
- `publishArticle` 因为 JSON 配置中 `idempotent: true`，使用了 `createIdempotentRequest()` 创建幂等请求
- `getArticles` 支持 `pager: true`，自动生成带有分页参数的请求函数

### 5.4 Demo 项目结构

以下是 `request-bus` 的实际目录结构，体现了样板代码与补丁代码的组织方式：

```
request-bus/
  index.ts          # 统一出口，合并样板与补丁
  templet/          # 自动生成的样板代码（勿手动修改）
    index.ts        # 汇总导出所有样板模块
    article.ts      # 文章相关的样板代码
    user.ts         # 用户相关的样板代码
  patch/            # 手动编写的补丁代码（覆盖样板）
    index.ts        # 汇总导出所有补丁
    article.ts      # 文章相关的补丁代码
```

### 5.5 模块代码详解

#### request-bus/index.ts —— 统一出口

```typescript
export * from './templet';

export { publishArticle } from './patch';
```

**代码解读：**

- 先导出 `templet` 目录下所有的样板代码
- 再从 `patch` 目录导出补丁函数，由于同名导出会覆盖前面的，补丁中的 `publishArticle` 会覆盖样板中的同名函数
- 这是 JavaScript 模块系统中**后导出覆盖前导出**的特性，巧妙实现了补丁机制

#### request-bus/templet/index.ts —— 样板代码汇总

```typescript
export * from './article';
export * from './user';
```

**代码解读：**

- 将所有样板模块统一汇总导出
- 新增业务模块（如 `order.ts`）时只需在此处追加一行即可

#### request-bus/templet/article.ts —— 文章样板代码

```typescript
export const publishArticle = () => {
  console.log('templet publishArticle');
};

  console.log('templet deleteArticle');
```

**代码解读：**

- 自动生成的样板代码，每个函数对应接口平台中定义的一个 API 端点
- `publishArticle` 和 `deleteArticle` 分别对应发布文章和删除文章接口
- 此文件由工具自动生成，不应手动修改

#### request-bus/templet/user.ts —— 用户样板代码

```typescript
export function regUser() {
  console.log('templet regUser');

```

**代码解读：**

- 用户模块的样板代码，定义了注册用户接口的请求函数

#### request-bus/patch/index.ts —— 补丁汇总

```typescript
export * from './article';
```

**代码解读：**

- 汇总所有补丁模块的导出
- 只有需要覆盖的模块才在此处引入

#### request-bus/patch/article.ts —— 文章补丁代码

```typescript

```

**代码解读：**

- 补丁文件中仅包含需要手动覆盖的函数
- 此处的 `publishArticle` 会在 `request-bus/index.ts` 中覆盖样板代码中的同名函数
- 补丁机制保证了对特殊接口的灵活处理，无需修改自动生成的样板代码

#### request-bus/test.ts —— 测试文件


**代码解读：**

- 测试文件验证补丁机制是否生效
- `publishArticle` 应输出 `patch publishArticle`（补丁覆盖了样板）
- `deleteArticle` 应输出 `templet deleteArticle`（没有补丁，使用样板）

## 六、补丁机制详解

### 6.1 设计思路

样板代码由工具自动生成，可能不满足所有业务需求。补丁机制提供了一种灵活的覆盖方式：

1. `templet/` 目录存放自动生成的样板代码（不可手动修改）
2. `patch/` 目录存放手动编写的补丁代码
3. `index.ts` 先导出所有样板代码，再导出补丁代码
4. 补丁中的同名函数会覆盖样板中的函数

### 6.2 覆盖流程

```
import { publishArticle } from 'request-bus'
         |
         v
  request-bus/index.ts
         |
         +-- export * from './templet'        --> 样板的 publishArticle
         |
         +-- export { publishArticle } from './patch'  --> 覆盖为补丁版本
         |
         v
  最终得到补丁版本的 publishArticle
```

### 7.2 面试话术

**请讲讲你是如何实现请求库的**

> **方案选择**
>
> 首先我考虑的是实现的必要性。前端本身存在诸多的请求库，像成熟度比较高的 axios，但它们都是一些基础库，没有提供上层功能，比如请求并发、请求重试、请求幂等这些都没有。另外像一些上层的请求库倒是有这些功能，比如 VueRequest、SWR，但它们都和具体的框架绑定了，并且没有提供请求的基础功能。更重要的一点是，市面上的公共库不可能提供基于公司内部协议的封装。所以我不得不考虑重新实现基于公司内部业务的通用请求库。

> **技术实现**
>
> 在实现层面，我首先考虑的是请求库的整体结构。我把请求库分为三层：请求基础库负责发送请求的基本功能，请求核心库负责实现各种上层逻辑（如并发），最上层是请求业务库，负责在代码层面封装公司内部协议。
>
> 在结构上，考虑到请求基础库的具体实现方式可能有变动，我应用 DIP 原则，参考了后端的 IOC 和 DI，让核心库不依赖具体的请求基础库，核心库仅提供 TS 接口，请求基础库可以基于接口提供不同的实现方式，在业务库注入具体实现即可。
>
> 另外，上层业务库深度绑定公司内部的接口文档，接口数量非常庞大，为了减少开发和维护成本，我用 Node 实现了一个自动化工具，通过解析接口标准文档自动为每个接口生成请求样板代码，并且制定了一种可以基于样板代码打补丁的开发方式。

> **落地效果**
>
> 整个请求库为公司的业务开发带来 30% 的效率提升，并依靠自动生成的样板代码，减少了 50% 的接口联调时间。业务开发人员只需调用请求库的业务函数即可，无须关心内部的并发、幂等这些复杂问题。
