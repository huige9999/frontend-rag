# 国际化的技术方案

## 课程概述

本节课围绕前端国际化（i18n）的常见技术方案展开，对比不同方案的优缺点、适用场景，帮助团队在实际项目中进行合理的技术选型。

## 知识点总结

### 一、国际化方案分类

前端国际化的技术方案大致可分为以下几类：

#### 1. 静态站点方案——多套站点

- 为每种语言构建独立的站点或页面目录（如 `/en/`、`/zh/`、`/ja/`）
- 构建时通过工具（如 Next.js 的 i18n 路由、Gatsby 的多语言插件）生成多份静态资源
- **优点**：SEO 友好，页面加载速度快，部署简单
- **缺点**：维护成本高，内容更新需全量构建，语言增多时体积膨胀

#### 2. 运行时方案——动态加载翻译资源

- 页面运行时根据用户语言偏好动态加载对应的翻译文件（JSON / YAML）
- 代表库：`react-i18next`、`vue-i18n`、`angular-localize` 等
- **优点**：灵活度高，按需加载，支持热切换语言，适合 SPA 应用
- **缺点**：需要额外的网络请求，首屏加载可能受影响，SEO 需额外处理

#### 3. 编译时方案——代码转换

- 在构建阶段将翻译内容直接注入到代码中，生成不同语言版本的 bundle
- 代表工具：`babel-plugin-react-intl`、`lingui`、`formatjs` 编译模式
- **优点**：运行时零开销，Tree-shaking 友好，性能最优
- **缺点**：构建流程复杂，调试困难，语言切换需刷新页面

#### 4. 服务端渲染（SSR）方案

- 服务端根据请求头（`Accept-Language`）或 URL 路径返回对应语言的内容
- 代表框架：Next.js（i18n 内置支持）、Nuxt.js（nuxt/i18n 模块）
- **优点**：SEO 友好，首屏性能好，用户体验佳
- **缺点**：服务端复杂度增加，部署和运维成本较高

### 二、主流 i18n 库对比

| 特性 | react-i18next | vue-i18n | formatjs (Intl) | lingui |
|------|--------------|----------|------------------|--------|
| 框架 | React | Vue | 框架无关 | React |
| ICU MessageFormat | 支持 | 支持 | 原生支持 | 支持 |
| 运行时/编译时 | 运行时 | 运行时 | 均支持 | 编译时优先 |
| 按需加载 | 支持 | 支持 | 支持 | 支持 |
| 复数/性别 | 支持 | 支持 | 原生支持 | 支持 |
| 生态丰富度 | 高 | 高 | 中 | 中 |
| 包体积 | 较小 | 较小 | 较大（可按需） | 较小 |

### 三、翻译资源管理

- **文件格式选择**：JSON（最常用）、YAML、PO（gettext）、CSV
- **资源组织方式**：
  - 按语言分目录（推荐）：`locales/en/common.json`、`locales/zh/common.json`
  - 按模块分文件：`en/common.json`、`en/user.json`
- **翻译工作流**：
  - 提取待翻译文本（提取脚本 / CLI 工具）
  - 导出给翻译团队或翻译平台（Crowdin、Transifex、Phrase）
  - 翻译完成后导入回项目
  - 自动化 CI/CD 集成

### 四、关键技术点

#### 语言检测与切换

- 浏览器语言检测：`navigator.language`、`Accept-Language` 请求头
- URL 路径策略：`/en/about`、`/zh/about`
- 子域名策略：`en.example.com`、`zh.example.com`
- 查询参数策略：`?lang=en`
- Cookie / localStorage 持久化用户语言偏好

#### 文本处理要点

- **日期与时间**：使用 `Intl.DateTimeFormat` 进行本地化格式化
- **数字与货币**：使用 `Intl.NumberFormat` 处理千分位、小数位、货币符号
- **复数规则**：不同语言的复数形式差异（如阿拉伯语有 6 种复数形式）
- **文本方向**：RTL 语言（阿拉伯语、希伯来语）需设置 `dir="rtl"`
- **排版与长度**：不同语言文本长度差异大，UI 需具备弹性

---

### 五、技术选型建议

| 场景 | 推荐方案 |
|------|---------|
| 纯 SPA 应用（不需要 SEO） | react-i18next / vue-i18n 运行时方案 |
| 内容型网站（强 SEO 需求） | Next.js / Nuxt.js SSR 方案 |
| 静态文档/博客站点 | 多套静态站点方案 |
| 性能敏感型应用 | lingui / formatjs 编译时方案 |
| 大型团队协作 | 运行时方案 + 翻译平台集成 |

---

### 六、最佳实践

1. **尽早引入国际化**：项目初期就规划 i18n 架构，后期改造成本高
2. **禁止硬编码文本**：所有用户可见文本必须走翻译流程
3. **使用 ICU MessageFormat**：统一处理插值、复数、性别等语法
4. **翻译 key 命名规范**：采用 `模块.功能.描述` 的层级命名（如 `user.profile.edit_title`）
5. **Fallback 机制**：缺失翻译时回退到默认语言，避免显示空白或 key
6. **自动化测试**：集成 i18n 相关的 lint 规则和自动化测试
7. **持续集成**：将翻译检查纳入 CI 流程，防止遗漏翻译
